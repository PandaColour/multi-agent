# 接口规范

## ⚠️ 开发边界

**可修改的系统**：
- `cloudbank-LCS`（核心服务）：可修改内部逻辑、接口实现
- `gws`（网关）：可修改

**不可修改的系统**：
- `target-lps`（获客系统）：不可修改，其调用方式和流程编排固定
- `scheduler`（定时任务）：不可修改，仅触发核心服务
- `argus`（风控系统）：不可修改
- 行方系统：不可修改

**设计原则**：
- 需求变更时，通过修改核心服务内部逻辑实现
- 不能依赖修改获客系统的流程编排

## 定时任务与核心服务的交互

### 协议上传任务

| 项目 | 说明 |
|-----|------|
| 触发方式 | scheduler 定时触发 |
| 执行位置 | cloudbank-LCS 核心服务-协议模块 |
| 任务内容 | 1. 扫描签署成功的协议<br>2. 根据协议类型判断处理逻辑<br>3. 上传、作废或等待 |

**处理逻辑（按协议类型区分）**：

| 协议类型 | 条件 | 处理动作 |
|---------|------|---------|
| 人脸协议 | - | 直接上传 |
| 百行授权书 | - | 直接上传 |
| 征信授权书 | - | 直接上传 |
| **授信合同** | 授信拒绝 | **不上传，不作废** |
| **授信合同** | 授信通过+建档成功 | 上传协议到行方 |
| **授信合同** | 授信通过+建档失败 | 不上传，不作废 |
| **授信合同** | 授信通过+建档处理中 | 不上传，不作废 |
| **授信合同** | 授信通过+无额度记录 | 不上传，不作废 |

**注意**：
- 授信合同需先查询授信状态，授信拒绝直接不上传
- 授信通过后再查询建档状态决定是否上传
- **模块间只有查询接口，无其他动作**

## 获客系统与核心服务的交互

### 穿插签署各类协议+建档串联流程

**授信通过场景**：

```
获客系统(target-lps)
       │
       ├──调用签署──► 核心服务-协议模块(签署人脸协议) ──签署成功──►
       │
       ├──调用签署──► 核心服务-协议模块(签署百行授权书) ──签署成功──►
       │
       ├──调用签署──► 核心服务-协议模块(签署征信授权书) ──签署成功──►
       │
       │   [授信通过]
       │
       ├──调用签署──► 核心服务-协议模块(签署授信合同) ──签署成功──►
       │
       └──调用建档──► 核心服务-额度模块(额度建档) ──建档成功──► 流程结束
```

**授信拒绝场景**：

```
获客系统(target-lps)
       │
       ├──调用签署──► 核心服务-协议模块(签署各类协议，含授信合同) ──签署成功──►
       │
       └──授信拒绝──► 流程结束（不调用建档）
                            │
                            ▼
                     定时任务扫描
                            │
                            ├──查询授信状态=拒绝
                            │
                            └──不上传，不作废
```

- 获客系统是**串联流程**的系统
- 在授信流程中**穿插调用**各类协议签署（人脸协议、百行授权书、征信授权书、授信合同）
- **授信合同签署**：无论授信通过还是拒绝，都会在穿插阶段签署
- **授信通过**：调用额度建档 → 流程结束 → 定时任务判断建档成功后上传
- **授信拒绝**：不调用建档，流程结束 → 定时任务查询授信状态后不上传、不作废
- 额度建档在业务逻辑上应**100%成功**，失败仅因网络或异常校验
- 建档失败为终态，不重试
