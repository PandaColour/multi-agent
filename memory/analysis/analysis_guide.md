# 需求分析指南

本文档记录需求分析过程中的注意事项和常见误区，帮助避免重复犯错。

---

## 一、术语辨析清单

在分析需求前，先确认以下易混淆术语：

| 术语对 | 区别说明 | 涉及模块 |
|-------|---------|---------|
| 签署 vs 上传 | 签署=电子签名，上传=传给行方 | 协议模块 |
| 授信 vs 额度建档 | 授信=审批通过，建档=创建额度档案 | 授信模块/额度模块 |
| 获客系统串联 vs 定时任务触发 | 串联=流程编排，触发=后台任务 | target-lps / scheduler |

---

## 二、开发边界约束 ⚠️

### 可修改的系统

| 系统 | 说明 |
|-----|------|
| cloudbank-LCS（核心服务） | 可修改内部逻辑、接口实现 |
| gws（网关） | 可修改 |

### 不可修改的系统

| 系统 | 说明 |
|-----|------|
| target-lps（获客系统） | 不可修改，其调用方式和流程编排固定 |
| scheduler（定时任务） | 不可修改，仅触发核心服务 |
| argus（风控系统） | 不可修改 |
| 行方系统 | 不可修改 |

### 设计原则

1. **需求变更时，只能通过修改核心服务内部逻辑实现**
2. **不能依赖修改获客系统的流程编排**
3. **签署动作由获客系统发起，核心服务不能改变签署时机**

---

## 三、模块间约束 ⚠️

### 核心约束

| 模块A | 模块B | 约束 |
|-------|-------|------|
| 核心授信模块 | 核心协议模块 | **只有查询接口，无其他动作** |
| 核心协议模块 | 核心额度模块 | **只有查询接口，无其他动作** |

### 违反约束的典型错误

❌ **错误做法**：在协议模块中调用额度模块执行建档动作
✅ **正确做法**：协议模块只能查询额度状态，建档动作由获客系统调用

❌ **错误做法**：在协议模块中调用授信模块修改授信状态
✅ **正确做法**：协议模块只能查询授信状态，不能修改

## 四、流程阶段识别

当分析合同/协议相关需求时，需识别涉及哪个阶段：

### 阶段1：授信流程（获客系统串联）

```
获客系统串联调用：
授信申请 → 穿插签署各类合同 → 额度建档 → 流程结束
```

**特点**：
- 由获客系统(target-lps)串联调用
- 实时流程，用户参与
- 涉及：签署动作

### 阶段2：定时任务上传（核心服务处理）

```
定时任务处理：
扫描签署成功的合同 → 判断条件 → 上传行方
```

**特点**：
- 由定时任务(scheduler)触发
- 后台异步处理
- 涉及：上传动作

## 五、常见误区

### 误区1：混淆"签署"和"上传"

**错误表现**：把上传条件的变更理解为签署时机的变更

**正确做法**：
- 签署时机 → 看获客系统的流程编排
- 上传条件 → 看定时任务的判断逻辑

### 误区2：忽略"不影响其他类型"的说明

**错误表现**：认为变更影响所有合同类型

**正确做法**：
- 明确需求涉及的合同类型（授信合同 vs 其他合同）
- 在文档中明确标注"不受影响"的范围

### 误区3：假设技术实现细节

**错误表现**：把业务条件的描述误解为技术实现的变更

**正确做法**：
- 需求描述中涉及技术位置变更时，必须与用户确认
- 区分"业务条件"和"技术实现位置"
- 不要自己脑补技术方案

### 误区4：忽略模块间约束

**错误表现**：设计方案时让协议模块调用额度模块执行动作

**正确做法**：
- 设计前先检查模块间约束（只能查询，无其他动作）
- 如需跨模块动作，需通过获客系统串联

### 误区5：忽略开发边界

**错误表现**：假设可以修改获客系统的流程编排

**正确做法**：
- 只能修改核心服务(cloudbank-LCS)和网关(gws)
- 获客系统的流程不可变，需在核心服务内部想办法

## 六、需求分析 Checklist

分析需求时，按以下顺序确认：

### 1. 识别变更类型
- [ ] 签署时机变更？（❌ 获客系统不可修改，签署流程不变）
- [ ] 上传条件变更？（✅ 定时任务逻辑调整）
- [ ] 新增合同类型？
- [ ] 状态流转变更？

### 2. 检查模块间约束
- [ ] 是否涉及协议模块调用额度模块？
- [ ] 是否涉及协议模块调用授信模块？
- [ ] 如有调用，是否只有查询？

### 3. 识别涉及范围
- [ ] 涉及哪些合同类型？
- [ ] 哪些合同类型不受影响？
- [ ] 涉及哪些系统模块？（是否在开发边界内？）

### 4. 确认接口复用
- [ ] 是否已有类似接口可复用？
- [ ] 接口调用方是谁？

### 5. 绘制时序图
- [ ] 明确参与者角色
- [ ] 区分授信流程和定时任务流程
- [ ] 标注关键判断节点

### 6. 确认边界条件
- [ ] 成功场景
- [ ] 失败场景
- [ ] 异常场景处理

## 七、历史教训记录

### 2026-04-07：授信合同签署时机 vs 上传条件

**原始需求**：
> "授信额度合同变更为授信通过后调用额度建档节点进行签署"

**错误理解1**：签署时机从"穿插签署"变为"建档节点时签署"
**错误理解2**：需要修改获客系统的流程编排

**正确理解**：
- 签署流程不变（仍穿插签署）
- 只改上传条件：授信未通过的合同不上传

**根因**：
1. 混淆了"签署"和"上传"
2. 忽略了开发边界约束（获客系统不可修改）
3. 自己脑补了技术方案

**改进措施**：
1. 需求描述涉及技术位置变更时，必须与用户确认
2. 先检查开发边界，判断哪些系统可修改
3. 优先考虑通过修改核心服务内部逻辑实现需求

### 2026-04-07：模块间约束的重要性

**场景**：分析授信合同上传规则时

**错误做法**：设计让协议模块调用额度模块执行签署动作

**正确做法**：协议模块只能查询额度状态，签署动作由获客系统发起

**根因**：没有提前检查模块间约束

**改进措施**：
1. 设计方案前，先检查 memory/analysis/architecture.md 中的模块关系
2. 记住核心约束：模块间只有查询接口，无其他动作

### 2026-04-07：复用已有接口

**场景**：定时任务需要查询授信状态

**发现**：代码中已有调用获客系统查询授信状态的接口，可直接复用

**经验总结**：
1. 分析需求时，先确认是否已有类似接口可复用
2. 不要假设需要新开发接口
3. 询问用户"代码中是否已有相关接口"

## 八、快速参考

### 授信合同处理规则速查

| 场景 | 签署 | 上传 |
|-----|------|------|
| 授信通过 + 建档成功 | ✅ 穿插签署 | ✅ 上传 |
| 授信通过 + 建档失败 | ✅ 穿插签署 | ❌ 不上传，不作废 |
| 授信拒绝 | ✅ 穿插签署 | ❌ 不上传，不作废 |

```
核心服务内部：

│  授信模块    │◄────────────│  协议模块    │
└─────────────┘              └──────┬──────┘
                                    │ 查询
                                    ▼
                             ┌─────────────┐
                             │  额度模块    │
                             └─────────────┘
```
