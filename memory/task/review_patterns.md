# 子代理复核经验模式

## 复核价值

子代理复核（阶段二）能发现初步拆分（阶段一）中遗漏的关键信息：
- 源码级别的路径覆盖分析（如双路径遗漏）
- 类名/方法名的实际验证（修正记忆文件中的错误描述）
- 实现方案的可行性评估和技术建议
- 风险点的提前识别

## 常见复核发现类型

### 1. 路径覆盖遗漏
初步拆分基于需求文档分析，可能只注意到某一条代码路径。子代理阅读源码后往往发现还有其他路径需要同步修改。

**案例（TASK-001）**：初步拆分只注意到 AuthFillJob 的降级路径（LPS fallback）需要修改，子代理发现主路径（核心查询成功分支）同样需要覆盖。

**预防策略**：任务拆分时明确标注"需全路径分析"，子代理复核时验证所有分支。

### 2. 描述与代码不符
记忆文件或需求分析中的类名、方法名、状态描述可能与实际代码不一致。

**案例（TASK-001）**：记忆文件记录 partner 层存在 AXQSignatureServiceImpl 和 JKYSignatureServiceImpl（注释状态），实际子代理确认这两个类不存在，真实类是 HkPartnerSignatureServiceImpl 和 PartnerSignServiceImpl。

**预防策略**：子代理复核发现描述不准确时，必须立即修正记忆文件。

### 3. 防重方案细化
初步拆分可能提出多种防重方案，子代理根据代码上下文给出明确的推荐和理由。

**案例（TASK-001）**：初步提出"接受重复查询"或"remark标记"两个方案，子代理明确推荐方案A（接受重复查询）作为第一阶段方案，并给出升级条件。

**预防策略**：防重方案在任务描述中给出推荐方案和升级路径，而非仅列出选项。

### 4. 待确认项识别
子代理可能发现某些行为依赖外部接口的具体返回值，需要在任务中标注为"上线前必须确认"项。

**案例（TASK-001）**：子代理发现主路径的修改取决于 queryCreditReq() 在授信拒绝时的返回行为（返回失败 vs 返回成功），需在实现前确认。

**预防策略**：不确定的接口行为不应假设，应在任务中明确标注为待确认项。

### 5. DTO 字段缺失导致方案变更
子代理阅读源码后可能发现，某条路径使用的 DTO 缺少关键字段，导致无法用统一方式判断，需额外调用其他接口补偿。

**案例（TASK-001）**：初步拆分假设主路径可用 ApplStatusEnum 判断授信状态，子代理发现主路径使用的 CreditReqQueryOutput（定义在 cloudbank-api）只有 status 字段、无 applState 字段。最终采用方案A：主路径额外调用 applQueryTargetFacade 获取 applState，与降级路径保持一致判断标准。

**预防策略**：涉及多条路径使用不同 DTO 做相同判断时，拆分阶段应主动确认各 DTO 的字段完整性，避免子代理复核后才发现方案需要调整。

## 复核反馈处理原则

1. **优先修正事实性错误**：类名、方法名、状态码等硬信息必须立即修正
2. **补充遗漏的修改点**：将子代理发现的额外代码路径纳入任务描述
3. **采纳技术方案建议**：子代理的方案建议通常基于实际代码分析，优先采纳
4. **保留待确认项**：不确定的结论不盲目假设，保留在任务中作为确认项
