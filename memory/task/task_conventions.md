# 任务拆分规范

## 拆分原则

1. **每个模块只分配一个任务**：将模块内所有需要完成的工作汇总为一条完整任务描述
2. **任务描述需包含**：
   - 优先级（P1/P2/P3）
   - 预估工时
   - 依赖关系
   - 具体实现步骤（定位代码 → 修改逻辑 → 适配扩展 → 编写测试）
3. **module 字段必须与 subagent.json 中的 name 一致**
4. **输出格式**：纯 JSON 数组，无 markdown 标记

## 任务描述模板

```
{功能名称}。优先级：{P1/P2/P3}。预估工时：{X天}。依赖：{依赖模块或无}。具体内容：
1. 定位现有逻辑代码（入口、判断条件、相关方法）
2. 修改/新增逻辑
3. 检查机构层扩展适配
4. 编写单元测试覆盖各用例场景
```

## 优先级说明

- P1：核心业务变更，影响主流程
- P2：功能增强，不影响主流程
- P3：优化调整，非紧急

## 用例覆盖要求

- 每个功能变更必须覆盖所有用例场景
- 在任务描述中明确列出用例编号和预期结果

## 拆分经验总结

### 1. 共用方法需明确修改层级
当多个 Job 共用一个 Service 方法时，任务描述中必须明确：
- 修改位置在**调用方（Job 层面）**还是**被调用方（共用方法内）**
- 明确标注"禁止修改共用方法"的约束，防止子代理误改影响其他流程

### 2. 定时任务场景需考虑防重循环
涉及定时任务拾取+跳过的逻辑时，必须考虑：
- 跳过的记录是否会被下次调度重复拾取
- 明确防重方案（remark 标记、新增状态、或接受重复查询）
- 如需修改 SQL 查询条件，需在任务中明确指出

### 3. 需求描述与代码实际的差异需核实
需求文档中的状态描述可能与代码实际状态不一致（如"签署成功状态"实际对应"CN未初始化状态"），复核后需在任务描述中修正，避免子代理困惑

### 4. 机构层检查结论需明确
机构层检查后无论是否需要适配，都应在任务中明确结论，避免子代理重复排查

### 5. 双路径/多分支逻辑需全路径覆盖分析
当代码存在主路径和降级/fallback路径时（如 AuthFillJob 的核心查询主路径+LPS降级路径），修改判断逻辑时必须检查所有路径的覆盖情况：
- 初步拆分时可能只注意到某一处分支，子代理复核往往能发现遗漏路径
- 任务描述中应明确标注"需同时覆盖主路径和降级路径"
- 如存在不确定性（如接口返回行为未确认），应作为"上线前必须确认"项写入任务

### 6. 记忆文件中的类名/状态描述需与代码实际一致
记忆文件中记录的类名、状态、方法等信息必须是子代理复核确认过的实际代码，不能用"注释状态""未启用"等模糊描述。如 partner 层实际签署类为 HkPartnerSignatureServiceImpl 和 PartnerSignServiceImpl，而非之前误记的 AXQSignatureServiceImpl/JKYSignatureServiceImpl。发现描述不准确时必须立即修正记忆文件。

### 7. 子代理复核后重新拆分的关注点
根据子代理复核反馈重新拆分任务时，重点关注：
- 补充子代理发现的遗漏修改点（如新发现的代码路径）
- 修正不准确的描述（如不存在的类名、错误的状态判断）
- 采纳子代理的方案建议（如防重方案的选择和理由）
- 保留"上线前必须确认"的待定项，不盲目假设结论

### 8. DTO 字段限制需提前识别
当多条代码路径需要做相同判断但使用不同 DTO 时，需提前确认 DTO 字段是否足够：
- 某条路径的 DTO 可能缺少另一条路径 DTO 中存在的关键字段（如 CreditReqQueryOutput 无 applState）
- 此时需额外调用其他接口获取缺失字段，或采用替代判断方案
- 任务描述中应明确指出 DTO 字段限制和补偿方案

### 9. 任务描述应包含具体代码定位信息
子代理复核反馈中如果给出了具体的文件名、方法名、行号范围，重新拆分时应将这些信息纳入任务描述：
- 具体行号（如 L121-128、L130-141）帮助子代理快速定位
- 明确每处修改点的当前行为和期望行为
- 减少"搜索相关代码"的时间开销
